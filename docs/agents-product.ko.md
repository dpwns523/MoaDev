# 제품용 에이전트 역할

## 목적

이 문서는 `MoaDev` 첫 릴리즈에서 필요한 논리적 에이전트 역할을 정의한다. 여기서 말하는 역할은 제품 책임 구분이지, 역할마다 별도 배포 서비스를 반드시 만들라는 뜻은 아니다.

목표는 `services/agents-runtime`이 제품 계약과 일치하도록 유지하면서도, 초기 구현은 작게 가져가는 것이다.

## 원칙

- 웹 클라이언트는 LLM을 직접 호출하지 않는다.
- API는 인증된 제품 화면과 처리 상태를 제공한다.
- `services/agents-runtime`은 비동기 수집 및 AI 가공 작업을 담당한다.
- 역할은 책임 기준으로 분리하되, 첫 릴리즈에서는 하나의 런타임 서비스 안에 함께 있을 수 있다.

## 에이전트 역할

### 1. 소스 수집 에이전트

책임:

- 승인된 기사 소스 항목을 발견하거나 수신한다
- 정책상 허용된 기사 메타데이터와 본문을 가져온다
- 소스 수집 상태와 업스트림 식별자를 기록한다

산출물:

- 소스 레코드
- 기사 메타데이터
- 원문 스냅샷 또는 포인터

### 2. 정규화 에이전트

책임:

- 가져온 콘텐츠를 정규화된 기사 구조로 변환한다
- 이후 번역이 가능하도록 본문을 안정적인 문단 또는 줄 단위로 분리한다
- 지원하지 않는 마크업이나 노이즈를 제거한다

산출물:

- 정규화된 기사 본문
- 순서가 보존된 콘텐츠 세그먼트
- 정규화 상태 및 경고

### 3. 지식 가공 에이전트

책임:

- 줄 단위 한국어 번역을 생성한다
- 간결한 요약을 생성한다
- 용어 설명을 생성한다
- 개념 설명을 생성한다
- 관련 개념을 제안한다

산출물:

- 번역 세그먼트
- 요약 데이터
- 용어집 데이터
- 개념 설명 데이터
- 관련 개념 데이터

### 4. 분류 및 게시 에이전트

책임:

- 카테고리와 태그를 부여한다
- 게시 가능한 수준으로 구조화 산출물이 충분한지 검증한다
- 레코드를 `published`, `needs_review`, `failed` 중 하나로 마킹한다
- API 소비자가 신뢰할 수 있는 최종 구조를 확정한다

산출물:

- 카테고리와 태그
- 게시 상태
- API 전달 준비가 된 기사 상세 데이터

## 엔드 투 엔드 파이프라인

```mermaid
flowchart LR
    source["승인된 소스"] --> intake["소스 수집 에이전트"]
    intake --> normalize["정규화 에이전트"]
    normalize --> enrich["지식 가공 에이전트"]
    enrich --> publish["분류 및 게시 에이전트"]
    publish --> api["services/api"]
    api --> web["apps/web"]
```

## API 경로와 런타임 경로 구분

### 동기 API 경로

- 사용자 인증
- 카테고리, 소스, 기사 조회 응답 제공
- 아직 완료되지 않은 기사의 처리 상태 노출
- 사용자 요청 경로 안에서 전체 기사 AI 가공을 수행하지 않음

### 비동기 런타임 경로

- 소스 fetch
- 콘텐츠 정규화
- 번역 및 설명 생성
- 분류
- 재시도 및 실패 큐 처리

## 구조화 산출물 계약

런타임은 다음 정보를 포함한 안정적인 기사 상세 레코드를 게시해야 한다.

- 기사 식별자와 소스 메타데이터
- 카테고리와 태그 데이터
- 정규화 세그먼트
- 세그먼트 정렬 한국어 번역
- 요약
- 용어집
- 개념 설명
- 관련 개념
- 처리 상태
- 부분 완료 또는 품질 저하 시 품질 메모

## 상태 모델

첫 릴리즈 권장 상태:

- `pending_intake`
- `pending_normalization`
- `pending_enrichment`
- `published`
- `needs_review`
- `failed`

## 재시도와 멱등성

- 기사 처리는 소스 이름과 canonical URL 같은 안정적인 식별자를 기준으로 키를 잡아야 한다.
- 같은 기사를 재처리할 때는 중복 레코드를 새로 만드는 대신 기존 논리 레코드를 갱신해야 한다.
- 수집, 정규화, 가공 단계는 서로 독립적으로 재시도 가능해야 한다.
- 영구 실패는 운영자가 볼 수 있는 이유와 함께 `needs_review` 또는 `failed`로 이동해야 한다.

## 이 문서가 하지 않는 것

- 첫 릴리즈에서 역할별 독립 배포 마이크로서비스를 강제하지 않는다
- 정확한 프롬프트 세트나 모델 라우팅까지 정의하지 않는다
- 제한 없는 업스트림 크롤링을 허용하지 않는다
