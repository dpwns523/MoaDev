## Overview

MoaDev uses Claude Code as its primary AI coding environment. The configuration is split into layers:

| Layer | Location | Purpose |
|-------|----------|---------|
| Project instructions | `CLAUDE.md` | Loaded automatically at session start — core principles, tech stack, workflow rules |
| Subagents | `.claude/agents/*.md` | Specialized roles spawned via the Task tool for domain-specific work |
| Slash commands | `.claude/commands/*.md` | Shortcut workflows invoked with `/command` in Claude Code |
| Skills | `.agents/skills/*/SKILL.md` | Detailed workflow playbooks read on demand |
| Settings | `.claude/settings.json` | Bash permission allowlist, environment variables |
| Claude reference | `docs/reference/ecc-claude-AGENTS.md` | Claude-specific supplemental guidance |

The design principle is **agent-first, skill-driven**: Claude Code reads `CLAUDE.md` on every session, then delegates specialized tasks to focused subagents or activates skill playbooks rather than doing everything inline.

## How It Works

### Session Start

When you open Claude Code in this project, it automatically reads `CLAUDE.md`. This gives Claude the full context it needs: tech stack, available subagents, skills, coding standards, and workflow rules. No manual setup required per session.

### Subagent Spawning

Claude uses the **Task tool** to spawn subagents defined in `.claude/agents/`. Each subagent is isolated — it starts with a fresh context and only the instructions in its `.md` file. This keeps expensive reasoning focused:

```
User: "Add email verification to the auth flow"
  └─ Claude spawns: explorer  →  reads auth code, reports findings
  └─ Claude spawns: docs-researcher  →  verifies NextAuth v5 API
  └─ Claude implements (guided by explorer findings)
  └─ Claude spawns: reviewer  →  checks correctness + security
  └─ Claude runs: /verify  →  build, lint, test, coverage gate
```

### Skill Activation

Skills are playbooks. Claude reads the relevant `SKILL.md` file before executing a complex workflow. This is how institutional knowledge (the right way to do TDD, the security checklist, etc.) is encoded once and reused consistently.

Slash commands are thin wrappers around skills — `/tdd` tells Claude to read `tdd-workflow/SKILL.md` and follow it.

## Development Flows

### Flow 1: New Feature

```
1. Open Claude Code → CLAUDE.md auto-loaded
2. Describe feature to Claude
3. Claude spawns explorer → maps relevant files, traces data flow
4. Claude spawns docs-researcher (if external API involved)
5. /tdd → write failing tests first (RED)
6. Implement until tests pass (GREEN)
7. Refactor while keeping tests green
8. Claude spawns reviewer → severity-rated findings returned
9. Fix CRITICAL/HIGH issues
10. /verify → build + typecheck + lint + test + security scan
11. Commit: feat: <description>
12. Open PR
```

### Flow 2: Bug Fix

```
1. Reproduce the bug and describe it to Claude
2. Claude spawns explorer → trace execution path to find root cause
3. /tdd → write a failing test that captures the bug (RED)
4. Fix the bug → test passes (GREEN)
5. Claude spawns reviewer → check for related regressions
6. /verify → confirm nothing else broke
7. Commit: fix: <description>
```

### Flow 3: Refactoring

```
1. Describe what to clean up
2. Claude spawns explorer → identify all usages and dependencies
3. /tdd → ensure existing tests cover the code being changed
4. Refactor in small, verifiable steps
5. Run tests after each step
6. Claude spawns reviewer → check for behavioral regressions
7. /verify → final gate
8. Commit: refactor: <description>
```

### Flow 4: Infrastructure / Platform Change

```
1. Describe the infrastructure change
2. /platform → Claude spawns platform-engineer subagent
3. Platform engineer reviews: blast radius, Helm correctness,
   Argo CD wiring, Terraform plan, rollback safety
4. Apply changes based on review findings
5. Re-run /platform to verify
6. Commit: chore: <description> or feat: <description>
```

### Flow 5: Release Preparation

```
1. "Prepare release vX.Y.Z"
2. Claude spawns release-manager subagent
3. Release manager checks: CI status, image tagging, Helm values
   promotion, release notes completeness, rollback plan
4. Work through the release checklist
5. Tag and push: git tag vX.Y.Z && git push origin vX.Y.Z
```

### Flow 6: Security Audit (Before Any PR)

```
1. /security → scans for hardcoded secrets, injection vectors,
   missing auth checks, dependency vulnerabilities
2. Fix all CRITICAL findings before opening PR
3. Verify with /verify
```

## Subagents Reference

All subagents are read-only by default — they report findings and do not make changes unless explicitly instructed.

### `explorer`
- **When**: Before any code change. Maps files, traces execution paths, identifies dependencies.
- **Model**: claude-sonnet-4-6
- **Returns**: Structured evidence report with file:line citations

### `reviewer`
- **When**: After writing or modifying code. Before opening a PR.
- **Model**: claude-sonnet-4-6
- **Returns**: Severity-rated findings (CRITICAL / HIGH / MEDIUM / LOW)

### `docs-researcher`
- **When**: Before implementing external API integrations or using unfamiliar framework features.
- **Model**: claude-sonnet-4-6
- **Returns**: Cited findings from primary documentation with version notes

### `platform-engineer`
- **When**: Helm, Argo CD, Terraform, or Kubernetes changes.
- **Model**: claude-sonnet-4-6
- **Returns**: Blast radius assessment + severity-rated infra review

### `observability-reviewer`
- **When**: Adding metrics, logs, traces, alerts, or dashboards.
- **Model**: claude-sonnet-4-6
- **Returns**: Missing instrumentation report + alert quality assessment

### `release-manager`
- **When**: Release preparation, CI workflow changes, GitOps promotion.
- **Model**: claude-sonnet-4-6
- **Returns**: Release checklist status + blocking issues

## Slash Commands Reference

| Command | Activates Skill | What It Does |
|---------|----------------|--------------|
| `/tdd` | `tdd-workflow` | Write failing tests → implement → verify ≥ 80% coverage |
| `/verify` | `verification-loop` | Build → typecheck → lint → test → security scan → diff review |
| `/security` | `security-review` | Full security checklist: secrets, input validation, auth, rate limiting |
| `/platform` | — | Spawn platform-engineer subagent for infra review |

### Usage Examples

```
/tdd            → start TDD for whatever feature is being discussed
/verify         → run before opening a PR
/security       → run before merging anything touching auth or secrets
/platform       → run after modifying anything in platform/ or infra/
```

## Skills Reference

Skills live in `.agents/skills/`. Each skill has a `SKILL.md` that Claude reads before executing the workflow. Skills are tool-agnostic — the same `SKILL.md` works for both Claude Code and Codex CLI.

| Skill | Trigger | Core Output |
|-------|---------|-------------|
| `tdd-workflow` | `/tdd`, new feature, bug fix | Passing tests with ≥ 80% coverage |
| `security-review` | `/security`, auth changes | Security checklist results |
| `verification-loop` | `/verify`, pre-PR | Build/test/lint/security report |
| `frontend-patterns` | Component/page work | React 19 + Next.js 16 best practices |
| `backend-patterns` | FastAPI endpoint work | Repository pattern, Pydantic, async handlers |
| `api-design` | New REST endpoints | Consistent envelope, versioning, OpenAPI |
| `python-patterns` | Python service work | Type hints, error handling, logging |
| `python-testing` | pytest test writing | Unit + integration test patterns |
| `database-migrations` | Schema changes | Alembic migration patterns |
| `e2e-testing` | Critical user flows | Playwright test patterns |
| `deployment-patterns` | Deploy changes | Deploy safety checklist |
| `docker-patterns` | Dockerfile changes | Multi-stage build, security |
| `coding-standards` | Code quality review | Universal naming, organization, immutability |
| `issue-driven-planning` | Complex features | GitHub issue breakdown before coding |
| `eval-harness` | Agent eval work | Eval-driven development workflow |

## Setup

### First-time setup (each developer, each machine)

```bash
# 1. Clone the repo — .claude/ is already included
git clone <repo-url>
cd MoaDev

# 2. Open Claude Code
claude

# Claude automatically reads CLAUDE.md and the agents/commands in .claude/ are ready.
```

No install script needed. `.claude/` is committed alongside the rest of the project.

### Updating the configuration

Edit the relevant file directly and commit:

```bash
# Example: add a new subagent
vim .claude/agents/my-new-agent.md
git add .claude/agents/my-new-agent.md
git commit -m "feat: add my-new-agent subagent"
```

### What gets committed vs. what stays local

| Path | Committed | Reason |
|------|-----------|--------|
| `CLAUDE.md` | ✅ Yes | Project-wide AI context |
| `.claude/agents/` | ✅ Yes | Shared subagent definitions |
| `.claude/commands/` | ✅ Yes | Shared slash commands |
| `.claude/settings.json` | ✅ Yes | Shared permission config |
| `.agents/skills/` | ✅ Yes | Shared workflow playbooks |
| `docs/reference/ecc-claude-AGENTS.md` | ✅ Yes | Claude-specific guidance |
